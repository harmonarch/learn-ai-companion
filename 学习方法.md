建议直接把学习方式切成这 5 层，效果会比只看教程快很多。

  1. 把 Claude Code 当成“陪练”，不要只当问答工具

  最有价值的用法不是“解释概念”，而是让它持续参与这几类动
  作：

  - 把抽象概念翻成最小可运行代码
    - 例如：
      - “用 30 行以内写一个最小 LangGraph agent”
      - “把 tool calling、state、router 各做一个最小例子”
  - 读你当前项目代码并指出问题
    - 例如：
      - “看下这里为什么状态没传下去”
      - “这个 graph 为什么会死循环”
  - 把教程内容映射到工程实践
    - 例如：
      - “这节讲的 memory，在真实 AI Agent
  产品里应该落在哪一层”
  - 反向出题检查理解
    - 例如：
      - “围绕 LangGraph 的 state reducer 给 3
  道题，偏工程判断题”
  - 做 code review
    - 例如：
      - “按 production 标准 review 这段 agent workflow”

  这样学，知识不会停在“看懂”，会进入“能写、能改、能判断”。

  ---
  2. LangChain 和 LangGraph 要分开学，不要混在一起

  很多人学乱，是因为把两者当同一层。

  LangChain 重点看什么

  LangChain 更适合当作：
  - LLM 调用封装
  - prompt 组织
  - output parser
  - tool binding
  - RAG pipeline
  - model / embedding / retriever 集成

  学习目标应该是：
  - 能快速组一个 单轮能力链
  - 能清楚每个组件输入输出是什么
  - 知道哪里适合封装，哪里不该过度封装

  LangGraph 重点看什么

  LangGraph 更适合当作：
  - agent workflow orchestration
  - state 管理
  - 节点跳转
  - 条件分支
  - 循环与终止
  - checkpoint / memory
  - multi-agent coordination

  学习目标应该是：
  - 能把一个 agent 过程画成 graph
  - 能判断什么时候该用 graph，什么时候一个 chain 就够
  - 能发现 workflow 中状态设计的问题

  一个很实用的顺序是：

  1. 先学 LangChain 的基础组件
  2. 再学 LangGraph 的状态流转
  3. 最后把两者拼起来做 agent

  否则很容易一上来就陷进“会调 API，但不知道 workflow
  为什么这么设计”。

  ---
  3. 学习路径要按“能力单元”推进

  建议按下面这个顺序学，每一块都要求 Claude Code
  深度参与。

  阶段 A：先打基础

  目标：知道 agent 系统最小构成

  建议依次掌握：

  1. LLM 调用
    - prompt
    - structured output
    - message format
  2. Tool Calling
    - tool schema
    - 什么时候模型决定调用工具
    - 工具结果怎样回流给模型
  3. Memory / State
    - 短期上下文
    - 会话状态
    - 持久化信息
  4. RAG
    - 检索
    - 重排序
    - 上下文注入

  这一步 Claude Code 的最佳角色是：
  - 给最小 demo
  - 给反例
  - 帮你比较不同实现的边界

  例如可以直接这样问：

  - “用 LangChain 写一个最小 tool calling demo，并解释
  message 是怎么流动的”
  - “给一个错误示例，说明什么时候不该把信息塞进 prompt”
  - “对比会话 memory 和业务 state，有什么本质差异”

  ---
  阶段 B：进入 LangGraph

  目标：理解 agent 不是“一次调用”，而是“可控制流程”

  重点学这几个点：

  1. StateGraph 的 state 设计
  2. node 的职责边界
  3. conditional edges
  4. 循环退出条件
  5. checkpoint / resume
  6. human-in-the-loop

  这一块最容易学浅。

  因为看教程时容易觉得：
  - node 就是函数
  - edge 就是跳转
  - state 就是个 dict

  但真正关键的是：
  - 哪些数据应该进 state
  - 哪些数据只是节点内部临时变量
  - 哪些字段会被后续节点依赖
  - 哪些循环会让 graph 无法终止
  - 哪些路由条件会让行为不可预测

  这类问题特别适合让 Claude Code 跟着你的代码一起分析。

  例如：

  - “帮忙检查这个 LangGraph state 设计有没有冗余字段”
  - “为什么这个 router node 容易把流程引到错误分支”
  - “把这个线性 agent 重构成 graph，但节点职责尽量清楚”

  ---
  阶段 C：用真实 Agent 场景练

  不要只做 hello world。

  建议尽快进入这 4 类项目，价值最高：

  1. Research Agent

  能力点：
  - 搜索
  - 检索
  - 总结
  - 引用来源
  - 多步推理

  2. Tool-use Agent

  能力点：
  - 调工具
  - 读工具结果
  - 错误恢复
  - 重试策略

  3. RAG Agent

  能力点：
  - query rewrite
  - retrieval
  - answer grounding
  - hallucination 控制

  4. Companion / Memory Agent

  能力点：
  - 用户画像
  - 长短期记忆
  - 对话状态
  - 情绪/角色一致性

  你现在这个方向本身就很适合学 LangGraph，因为 AI
  companion
  本来就是一个长期状态系统，天然不是“单次问答”产品。

  4. 让 Claude Code 深度参与，最好固定成 6 种模式

  这部分最关键。每次学习都可以明确指定一种模式。

  模式 1：Teacher

  用途：讲概念
  提示词例子：

  - “把 LangGraph 的 state、node、edge
  用后端系统设计视角讲清楚”
  - “讲 tool calling，不要只讲定义，要讲执行链路”

  模式 2：Pair Programmer

  用途：边写边学
  提示词例子：

  - “和当前项目一起写一个最小 LangGraph
  agent，每一步先说目的再改代码”
  - “不要一次性写完，按节点逐步实现”

  模式 3：Debugger

  用途：查 bug
  提示词例子：

  - “这个 agent 进入死循环了，定位根因并给最小修复”
  - “看日志和 graph
  定义，判断问题出在状态更新还是路由条件”

  模式 4：Reviewer

  用途：提升工程判断
  提示词例子：

  - “review 这段 LangChain/LangGraph
  代码，重点看状态设计、扩展性和错误调用路径”
  - “按 production 视角指出最值得先修的 3 个问题”

  模式 5：Examiner

  用途：检查掌握程度
  提示词例子：

  - “围绕 LangGraph 的 conditional edges 出 3
  道题，偏场景判断”
  - “出题后不要立刻给答案，等作答后再批改”

  模式 6：Architect

  用途：从 demo 走向产品
  提示词例子：

  - “把当前单 agent demo 演进成 companion
  agent，需要哪些模块”
  - “哪些应该放进 graph，哪些应该留在 graph 外部服务层”

  5. 学 LangChain / LangGraph
  时，最该刻意训练的是这几个能力

  只学 API 很容易停留在表面。更值得训练的是：

  A. 画执行流

  看到一个 agent，要能说清：

  - 输入从哪里来
  - 哪一步决定是否调用工具
  - 工具结果写回哪里
  - 状态在哪更新
  - 哪个条件决定下一跳
  - 什么时候结束

  B. 设计 state

  很多 LangGraph 项目，问题都不在模型，而在 state
  设计差。

  要经常追问：
  - 这个字段是谁写的
  - 谁会读它
  - 生命周期多长
  - 是业务状态还是推理中间态
  - 是否会污染后续节点判断

  C. 判断抽象层级

  例如：
  - 这个逻辑应该是一个 node，还是 node
  内部的一段普通函数？
  - 这个流程值得建 graph 吗？
  - 这个 memory 应该在 LangGraph
  checkpoint、数据库、还是应用层 session？

  D. 做失败路径设计

  Agent 开发的核心能力之一，是处理不顺利的路径：

  - 工具报错怎么办
  - 搜索结果为空怎么办
  - 模型没有按 schema 输出怎么办
  - 路由判断不稳定怎么办
  - 对话上下文太长怎么办

  这个部分最适合让 Claude Code 帮你模拟反例。

  ---
  6. 最推荐的实战学习法：每学一个点，都做“正例 + 反例 +
  改造题”

  比如学 tool
  calling，不要只做一个能跑的例子，要做三件事：

  正例

  - 正常调用天气工具并回答

  反例

  - 工具返回异常
  - 工具参数缺失
  - 模型没选工具却应该选工具

  改造题

  - 把单次 tool calling
  改成“先判断是否需要工具，再执行，再总结”的 graph

  这样学，才会形成迁移能力。

  8. Claude Code 在这个过程中最适合承担的“深度参与”方式

  这里给一个直接可执行的协作方式。

  每次学习一个知识点时，要求 Claude Code 同时做 4 件事：

  1. 解释概念
  2. 写最小代码
  3. 指出真实工程里的坑
  4. 给一道改造练习

  比如你可以固定这样提问：

  ▎ 讲 LangGraph 的 checkpoint。
  ▎ 要求：
  ▎ 1. 用工程视角解释
  ▎ 2. 给最小可运行例子
  ▎ 3. 指出 3 个常见误区
  ▎ 4. 基于当前项目给一个小练习

  这个提问方式比“给我讲讲 checkpoint”有效很多。

  ---
  9. 一个很关键的判断：什么时候该用
  LangChain，什么时候该用 LangGraph

  可以先记一个非常实用的标准。

  用 LangChain 的情况

  - 单轮调用为主
  - 流程简单
  - 更像 pipeline
  - 步骤固定
  - 不需要复杂状态流转

  用 LangGraph 的情况

  - 有多步决策
  - 有条件分支
  - 有循环
  - 有长期状态
  - 需要恢复、暂停、人工介入
  - 有多个 agent 或多个角色协作

  对于 AI companion，大概率后面会逐步走向
  LangGraph，因为它涉及：
  - 长会话
  - 状态积累
  - 记忆读写
  - 不同对话阶段的流程控制

  10. 当前最值得立刻开始的学习方法

  直接选一个非常小的目标：

  做一个“带工具调用 + 会话状态”的最小 companion agent

  只需要这几个能力：
  - 用户发消息
  - agent 判断要不要调工具
  - 工具返回结果
  - state 记录最近对话与用户偏好
  - 最终回复用户

  这个项目足够小，能学到：
  - LangChain 的 tool / model 绑定
  - LangGraph 的 state / node / edge
  - companion 场景里的 memory 雏形

  - 先定一个知识点

  - 故意制造一个 bug

  4. AI companion 里的 memory 应该怎样分层

  发一个数字就行：

  - 1：进入 LangChain tool calling
  - 2：进入 LangGraph state 设计
  - 3：进入 最小 LangGraph agent
