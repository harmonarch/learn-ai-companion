## 一、项目定位与业务目标

本项目是一个**企业级 AI 电子伴侣产品**，核心目标是让大语言模型（LLM）从"单轮问答工具"升级为"长期存在的关系型 Agent"。

**核心命题**：AI 的"智能"来自大模型本身，我们的工作是给模型补上它缺失的能力——记忆、情绪、安全、持久化、多用户管理。

---

## 二、五道工程鸿沟（从提示词到产品）

一段提示词就能让 LLM 扮演 AI 伴侣，但从"能用"到"产品"需要跨越五道工程鸿沟：

| # | 鸿沟 | 问题描述 | 解决方案 |
|---|------|---------|---------|
| 1 | **记忆丢失** ⭐ | LLM 无持久记忆，关闭窗口后一切归零 | RAG + 混合记忆架构（向量检索 + 结构化存储） |
| 2 | **上下文溢出** ⭐ | Token 窗口有限，无法装入所有历史 | Memory Scheduler 内存调度器 + 滑动窗口 + 压缩 |
| 3 | **情绪断裂** ⭐ | LLM 不追踪情绪状态，每轮回复"无状态" | 情绪状态机（Emotion FSM） |
| 4 | **安全失控** | 提示词注入、角色出戏、违规内容 | 输入输出安全检查、Middleware 层 |
| 5 | **无法运营** | 无法支持多用户、收费、迭代 | 完整工程化（认证鉴权、后台管理、CI/CD） |

> ⭐ 标记为**核心技术难点**

## 四、三子系统详解

### 4.1 Hono 服务端 —— AI 对话的中枢神经

服务端是核心，承担所有 AI 相关逻辑编排。**它不做推理，但决定推理的质量。**

- **对话管线**：安全检查 → 意图识别 → 情绪分析 → 记忆检索 → Prompt 组装 → LLM 调用 → 流式输出 → 记忆写回
- **用户认证与鉴权**
- **记忆系统 CRUD**
- **情绪状态管理**
- **后台管理接口**

### 4.2 Next.js 客户端 —— 用户直接触达的产品界面

- 聊天界面（核心体验）
- 用户注册/登录
- 个人设置
- SSR/SSG 落地页
- 记忆回顾功能

### 4.3 Next.js 后台管理 —— 运营与维护控制台

- 用户数据总览
- 对话审计与内容安全
- **Prompt 模板管理**（不改代码，只调 Prompt 即可改变 AI 行为）
- 系统配置
- 运维监控

> **为什么分两个 Next.js 应用？** 用户群体不同、安全边界不同、构建部署独立、包体积控制。

## 五、核心数据流（一条消息的完整链路）

```
用户发送消息
    │
    ▼
┌───────────────────────────────────────────────────┐
│ 边缘节点 (Cloudflare Worker)                       │
│                                                    │
│  1. 安全检查 ─────────────────── 5-20ms            │
│  2. 意图识别 ─────────────────── 10-30ms           │
│  3. 情绪分析 ─────────────────── 10-30ms           │
│  4. 记忆检索                                       │
│     ├─ 向量检索 (Vectorize) ──── 30-100ms          │
│     ├─ 关键词匹配 (D1) ──────── 10-30ms           │
│     └─ 情绪状态读取 (KV) ────── 1-5ms             │
│  5. Prompt 组装 ──────────────── 5-10ms            │
│  6. LLM API 调用 (首Token) ──── 500-2000ms ⚠️     │
│  7. 流式输出给用户                                  │
│  8. 记忆写回 (异步, P1/P2)                          │
│                                                    │
│  总 TTFB: 1-2 秒                                   │
└───────────────────────────────────────────────────┘
```

> ⚠️ **LLM 推理是不可压缩的硬成本** (500-2000ms)，因此其他环节的优化空间就是边缘部署的价值所在。

## 六、内存调度架构 ⭐ 核心难点

### 6.1 OS 类比

| 计算机体系 | AI 伴侣系统 |
|-----------|------------|
| CPU | LLM（大语言模型） |
| RAM | 上下文窗口（Token 限制） |
| 硬盘 | 向量数据库 + D1 结构化存储 |
| OS 内核 | Memory Scheduler 内存调度器 |
| 页置换算法 | 记忆检索 + 压缩 + 淘汰策略 |

### 6.2 任务优先级

| 级别 | 类型 | 说明 | 要求 |
|------|------|------|------|
| **P0** | 实时交互链路 | 用户消息 → 回复 | 必须"秒回" |
| **P1** | 关键状态更新 | 情绪更新、重要记忆写入 | 下轮对话前生效 |
| **P2** | 记忆整理优化 | 记忆合并、摘要生成 | 后台异步执行 |

### 6.3 五大调度场景

1. **长期记忆精准调用**（RAG 调度）—— 跨越数百轮对话召回关键信息
2. **情绪状态机"记仇"模式** —— 根据情绪状态动态调整回复风格
3. **话题切换与上下文压缩** —— 检测话题边界，压缩旧话题释放 Token
4. **多模态感知与工具调用** —— 判断何时需要外部工具介入
5. **主动关怀（Time-based Dispatch）** —— 基于时间触发的主动问候

## 七、LangChain 与 LangGraph 的协作关系 ⭐

```
┌─────────────────────────────────────────────────┐
│            LangGraph = 大脑与神经中枢             │
│                                                  │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐     │
│  │  Router  │───▶│ Memory  │───▶│  State  │     │
│  │ (意图判断)│    │(记忆检索) │    │(状态更新)│     │
│  └─────────┘    └────┬────┘    └────┬────┘     │
│       │              │              │           │
│       │         调用 LangChain  调用 LangChain   │
│       │              │              │           │
│       ▼              ▼              ▼           │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐     │
│  │Generator│    │Retriever│    │  Prompt  │     │
│  │(生成回复) │    │(向量检索) │    │ Assembly │     │
│  └─────────┘    └─────────┘    └─────────┘     │
│                                                  │
│  LangChain = 器官和肢体（原子能力执行者）          │
└─────────────────────────────────────────────────┘
```

- **LangGraph** 负责：全局状态管理、决策路由、持久化 (Checkpointer)、多 Agent 编排
- **LangChain** 负责：记忆检索、动态 Prompt 组装 (LCEL)、工具调用、Output 解析

> **核心洞察**：没有 LangChain，开发困难；没有 LangGraph，Agent 没有灵魂（逻辑混乱且记不住状态）。

## 八、边缘部署架构 ⭐

### 为什么选 CloudFlare Workers

| 维度 | 集群部署 | 边缘部署 (CF Workers) |
|------|---------|---------------------|
| 网络延迟 | 100-400ms | 10-50ms |
| 冷启动 | 100-3000ms | ~0ms |
| 扩容 | 需配置自动伸缩 | 自动，无需干预 |
| 运维 | 高（服务器、容器、监控） | 低（平台托管） |
| 全球覆盖 | 多区域部署成本高 | 310+ 节点自动就近 |

### AI 伴侣特别需要边缘部署的四个原因

1. **情感连接对延迟极度敏感** —— 延迟破坏情感沉浸
2. **高频短消息模式** —— 每条 20-50 字，一天 50-200 条
3. **全球用户无法靠"选区域"解决**
4. **流量波峰波谷明显** —— 不需要为"空房间"付电费

### 推荐方案

**方案 A + 边缘调度**：调度逻辑（安全检查、记忆检索、Prompt 组装）在边缘完成，LLM 推理调用第三方 API (DeepSeek/Claude/GPT)。

## 九、Monorepo 工程化架构

```
ai-companion/
├── apps/
│   ├── web/          # Next.js 客户端（用户聊天）
│   ├── admin/        # Next.js 后台管理
│   └── server/       # Hono.js 服务端（CF Workers）
├── packages/
│   ├── shared/       # 类型定义 / Zod Schema / 错误码
│   ├── ui/           # 共享 UI 组件
│   └── config/       # tsconfig / biome 配置
├── turbo.json        # Turborepo 任务编排
└── package.json      # workspace 定义
```

### 选择 Monorepo 的五大理由

1. **类型共享是第一驱动力** —— Hono RPC 端到端类型安全要求前后端同仓库
2. **共享业务逻辑** —— Zod Schema、错误码、工具函数
3. **原子提交保证版本一致** —— 消除跨仓库版本同步问题
4. **统一工具链** —— 一套 tsconfig / lint / format 配置
5. **即时反馈环路** —— 改后端 API，前端即刻感知类型变化

## 十二、技术重点与难点总结

### 🔴 高难度（需要深入理解）

1. **混合记忆架构设计** —— 短期记忆 + 长期记忆 + 语义检索 + 结构化查询的协调
2. **内存调度器 (Memory Scheduler)** —— 在有限 Token 窗口内动态选择最优上下文
3. **LangGraph 多 Agent 编排** —— Supervisor / Handoff / Swarm / 层级团队模式
4. **情绪状态机设计** —— 有限状态机 + 状态持久化 + 跨会话连续性
5. **RAG 管线优化** —— 检索质量、幻觉控制、混合检索策略

### 🟠 中等难度（需要实践经验）

6. **LangGraph StateGraph 与条件路由** —— 节点、边、Reducer、数据流控制
7. **Checkpointer 状态持久化** —— 时间旅行、断点恢复
8. **流式响应 (Streaming)** —— SSE 实现、首字节优化
9. **Hono.js + CF Workers 全家桶集成** —— D1 + KV + Vectorize + R2 协同
10. **Prompt 工程深度实践** —— 动态 Prompt、Few-Shot、角色一致性维护

### 🟡 基础但重要

11. **LCEL (LangChain Expression Language)** —— 并行、分支、容错
12. **Monorepo + Turborepo** —— 工程化落地、依赖管理、CI/CD
13. **可观测性与 Tracing** —— LangSmith / Langfuse 集成
14. **边缘部署策略** —— 延迟优化、CloudFlare 平台特性
