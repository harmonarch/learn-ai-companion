# React Query 学习计划

## 这个领域解决什么问题

React Query 负责前端异步数据获取、缓存、失效、变更同步和请求状态管理，适合处理 UI 与后端接口之间的数据协作问题。

## 学习目标

- 理解 query、mutation、cache、invalidate 的关系
- 能设计前端异步状态的最小结构
- 能解释一次数据请求从发起到缓存更新的链路
- 能把 React Query 接到 Hono RPC 或普通 API 上
- 能判断 React Query 和本地状态管理的边界

## 四阶段路径

### 阶段 1：入门理解
- 重点：query、mutation、query key、cache
- 产出：能写一个最小列表查询

### 阶段 2：核心能力
- 重点：失效、重取、乐观更新、错误状态
- 产出：能写一次 mutation 并联动刷新列表

### 阶段 3：实战落地
- 重点：与 Hono RPC、聊天历史、后台列表页结合
- 产出：能管理一个真实页面的数据请求状态

### 阶段 4：巩固与排障
- 重点：重复请求、缓存污染、query key 设计
- 产出：能说清数据没更新到底卡在哪一层

## 核心易混点

- React Query 管的是服务端状态，不是所有前端状态。
- invalidate、refetch、setQueryData 的使用时机不同。
- query key 设计不清会直接破坏缓存语义。
- 页面不刷新时，问题可能在请求、缓存，也可能在渲染层。

## 各阶段过关标准

- 阶段 1：能解释它解决的问题，并写出或画出一个最小结构。
- 阶段 2：能独立完成核心能力的小例子，并说清执行流。
- 阶段 3：能把它接到当前 AI 伴侣项目的上下游，并解释边界。
- 阶段 4：能围绕典型错误定位根因，给出最小修复和复盘。

## 必做失败演练

- 故意写不稳定的 query key，观察缓存为什么错乱。
- 故意让 mutation 成功后不做失效或本地更新，练习定位页面为什么不刷新。
- 故意做不完整的乐观更新，观察回滚缺失会怎样污染 UI。

## 改造练习

- 把一次手写请求改造成 query + mutation + 缓存失效方案，并补一套明确的 query key 设计。

## 系统连接

- 上游常接 Next.js 页面和组件状态。
- 下游常接 Hono RPC、普通 API、Zod 契约。
- 在 AI 伴侣里常落在前端数据协作层和后台/列表页状态层。

## 常见误区

1. 把所有状态都交给 React Query 管
2. query key 设计随意，导致缓存错乱
3. mutation 成功后不做失效或本地更新
4. 只会调用，不会解释缓存策略
