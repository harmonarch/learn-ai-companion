# Cloudflare 学习计划

## 这个领域解决什么问题

Cloudflare 提供 Workers、KV、D1、R2、Vectorize、Durable Objects、Queues、Cron 等边缘能力，负责承载 API、存储、异步任务、实时连接和 AI 基础设施。

## 学习目标

- 理解 Workers 运行时与传统 Node 服务的差异
- 能区分 KV、D1、R2、Vectorize、Durable Objects 的适用场景
- 能说清一次边缘请求从入口到存储再到响应的链路
- 能把 Hono、Drizzle、AI 能力部署到 Cloudflare 上
- 能围绕性能、延迟、存储选择做判断

## 核心易混点

- Workers、KV、D1、R2、DO、Queues、Cron 各自解决的问题不同。
- 边缘运行时和传统 Node 运行时能力边界不同。
- 存储选型要看一致性、延迟、结构和状态模式。
- 本地跑通不代表部署环境就一致。

## 各阶段过关标准

- 阶段 1：能解释它解决的问题，并写出或画出一个最小结构。
- 阶段 2：能独立完成核心能力的小例子，并说清执行流。

- 阶段 4：能围绕典型错误定位根因，给出最小修复和复盘。

## 必做失败演练

- 故意漏掉运行时绑定，练习定位是配置问题还是代码问题。
- 故意把聊天状态放到不合适的存储里，比较选型错误会带来什么后果。
- 故意在 Workers 中使用不合适的 Node 习惯，观察运行时限制如何暴露。

## 改造练习

- 把一个本地服务改造成部署到 Cloudflare Workers 的边缘服务，并重新拆分存储边界。

## 系统连接

- 上游常接 Hono、Drizzle、AI SDK、边缘日志与 telemetry。
- 下游常接 KV、D1、R2、DO、Queues 等基础设施。
- 在 AI 伴侣里常落在边缘运行时层和存储选型层。

## 常见误区

1. 把 KV、D1、R2 当成同类存储使用
2. 不考虑 Workers 运行时限制，直接搬 Node 心智
3. 没有明确数据和状态边界，选型混乱
4. 只会本地跑，部署后才发现绑定和环境问题
