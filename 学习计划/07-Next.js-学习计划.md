# Next.js 学习计划

## 这个领域解决什么问题

Next.js 负责前端页面、App Router、服务端渲染、Server Actions、RSC 与前后端整合，是 AI Chat、后台管理和流式 UI 的主要承载层。

## 学习目标

- 理解 App Router、Server Components、Client Components 的边界
- 能说清页面请求、数据获取、流式渲染的链路
- 能把 AI Chat UI 接到服务端接口
- 能理解 Next.js 与 Vercel AI SDK 的配合方式
- 能判断哪些逻辑该留在前端，哪些要放到 Hono 或服务端

## 四阶段路径

### 阶段 1：入门理解
- 重点：App Router、RSC、客户端与服务端组件边界
- 产出：能画出一个页面的渲染链路

### 阶段 2：核心能力
- 重点：聊天页面、输入输出渲染、流式消息消费
- 产出：能实现一个最小 AI Chat 页面

### 阶段 3：实战落地
- 重点：与 Hono、AI SDK、MCP 工具结果渲染整合
- 产出：能串起 UI → API → AI → UI 回流

### 阶段 4：巩固与排障
- 重点：水合问题、客户端状态、流式渲染问题
- 产出：能定位交互和渲染层故障

## 核心易混点

- Server Component、Client Component、Route Handler 不是同一层。
- 页面状态、服务端数据、流式消息状态要分开处理。
- Next.js 接口层和 Hono 接口层的职责边界要分清。
- 能显示页面，不代表已经看清渲染链路。

## 各阶段过关标准

- 阶段 1：能解释它解决的问题，并写出或画出一个最小结构。
- 阶段 2：能独立完成核心能力的小例子，并说清执行流。
- 阶段 3：能把它接到当前 AI 伴侣项目的上下游，并解释边界。
- 阶段 4：能围绕典型错误定位根因，给出最小修复和复盘。

## 必做失败演练

- 故意把必须放客户端的逻辑写进服务端组件，观察边界为什么出错。
- 故意制造水合不一致，练习怎样从渲染现象回推原因。
- 故意让流式消息只到服务端不更新 UI，定位问题卡在接口、状态还是渲染层。

## 改造练习

- 把一个静态页面改造成消费 AI 流式消息的聊天页面，并重新划分服务端和客户端职责。

## 系统连接

- 上游常接 React Query、Vercel AI SDK、前端状态管理。
- 下游常接 Hono、模型接口和工具结果渲染。
- 在 AI 伴侣里常落在页面层、流式 UI 层和交互展示层。

## 常见误区

1. 不区分 Server Component 和 Client Component 的职责
2. 把所有状态都堆到前端，导致服务端边界混乱
3. 只会渲染文本，忽略工具片段和对象流
4. 页面能跑，但解释不清请求链路和渲染时机
