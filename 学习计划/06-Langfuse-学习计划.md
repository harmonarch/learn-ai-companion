# Langfuse 学习计划

## 这个领域解决什么问题

Langfuse 提供 LLM 应用的 tracing、观测、Prompt 版本管理、评分和分析能力，适合做生产环境下的链路观察与效果追踪。

## 学习目标

- 理解 Langfuse 在 AI 应用观测中的定位
- 能把调用链路和业务评分关联起来
- 能区分 Langfuse 与 LangSmith 的使用重心
- 能围绕生产问题读 trace、看指标、看 prompt 版本
- 能把观测结果反过来改进系统

## 四阶段路径

### 阶段 1：入门理解
- Ch16-Ch17
- 重点：观测、评分、Prompt 版本的作用
- 产出：能说清 Langfuse 想解决哪类生产问题

### 阶段 2：核心能力
- Ch43 及相关 tracing 接入
- 重点：trace、score、prompt 管理、环境区分
- 产出：能把一条调用链完整记录下来

### 阶段 3：实战落地
- 结合 Ch100、Ch132
- 重点：生产环境信号、故障定位、版本回看
- 产出：能根据 trace 和 score 判断一次改动的影响

### 阶段 4：巩固与排障
- 重点：数据太多时如何聚焦、怎样围绕关键问题观察
- 产出：能从“看到很多数据”走到“知道该看哪几个信号”

## 核心易混点

- Langfuse 更偏生产观测和效果追踪，不只是看 trace。
- trace、score、Prompt 版本、业务指标要分开看。
- 环境区分和版本管理不到位时，观测数据会失真。
- Langfuse 和 LangSmith 侧重点不同。

## 各阶段过关标准

- 阶段 1：能解释它解决的问题，并写出或画出一个最小结构。
- 阶段 2：能独立完成核心能力的小例子，并说清执行流。
- 阶段 3：能把它接到当前 AI 伴侣项目的上下游，并解释边界。
- 阶段 4：能围绕典型错误定位根因，给出最小修复和复盘。

## 必做失败演练

- 故意只接 tracing 不接评分，观察为什么很难判断改动是否有效。
- 故意混用测试环境和生产环境数据，练习怎样识别噪音。
- 故意让评分规则过于模糊，比较分数为什么不具解释力。

## 改造练习

- 给一个最小 Agent 同时加上 trace、score 和 Prompt 版本，再用这些信息判断一次改动效果。

## 系统连接

- 上游常接 AI SDK、LangChain、LangGraph 等运行链路。
- 下游常接线上分析、Prompt 管理和效果评估。
- 在 AI 伴侣里常落在生产观测层和持续改进层。

## 常见误区

1. 只接 tracing，不接业务评分，导致看不到效果变化
2. 看到 trace 就停，不继续回到代码和 Prompt 做修正
3. 不做版本管理，之后很难追溯行为变化
4. 把 Langfuse 和 LangSmith 的边界混在一起
