# 学习方法与评估标准

这份文档适用于 `学习计划/` 下所有专项。
默认把 Claude 当成学习陪练来使用：不只解释概念，还要陪着把知识点讲清、写小例子、看失败路径、做调试和改造练习。

## 一、每轮开场先锚定

每次进入一个知识点，先确认这 4 项：

1. 当前章节或阶段
2. 当前目标
3. 当前卡点
4. 当前模式：讲解 / 写代码 / 查 bug / 审查 / 测验 / 架构判断

缺任意一项时，优先补齐，再进入主线。

## 二、单点推进节奏

推荐每次只推进一个知识点，固定按下面顺序走：

1. 先回答：它解决什么问题
2. 再讲：适用边界和不适用边界
3. 再写：最小可运行示例
4. 再拆：执行流 / 请求流 / 数据流
5. 再看：常见失败路径和排查顺序
6. 最后做：一道改造题、调试题或小练习

不要在同一轮里同时推进多个主线。

## 三、陪练协议

所有专项默认执行下面这套陪练动作：

- 先用当前章节语境解释，不脱离本仓库主线空讲概念
- 先给最小可运行版本，再逐步贴近真实系统
- 每轮至少补一个失败案例，说明错在哪、先查什么、最小修复是什么
- 每轮至少留一个动手任务，优先使用改造题或调试题
- 学员表述模糊时，先追问输入、输出、状态变化和边界选择，再继续讲
- 能映射到 AI 伴侣项目时，要说明它在完整产品里位于哪一层

## 四、如何配合 skill 学

每个专项都配有一个项目内 slash skill。

推荐提问方式：

- `解释 <知识点>，给最小示例，并说明执行流`
- `围绕 <知识点> 出 3 道场景题`
- `review 这段代码，重点看 <边界/状态/类型/存储>`
- `把当前实现改造成更贴近生产使用的版本`
- `定位这个 bug，说明根因和最小修复`
- `按当前章节继续，先讲最容易混淆的地方`

## 五、什么算学会

一个知识点达到“已掌握”，至少满足下面 4 条中的 3 条：

- 能用自己的话解释它解决的问题和边界
- 能写出最小可运行代码或最小结构
- 能说清输入、处理、输出、状态变化
- 能指出 2 个以上常见错误和排查思路

章节通过时，再额外满足这 2 条：

- 能把这个知识点接到当前 AI 伴侣项目语境里
- 能完成一道失败路径修复题或改造题

## 六、统一测验协议

当前仓库沿用数字协议：

- `1`：围绕当前知识点出 3 题
- `0`：换一种方式继续讲当前知识点
- `2`：回答临时问题，但不切换当前学习主线

测验题优先使用：

- 场景判断题
- 执行流推理题
- 架构边界选择题
- 调试题

少用纯名词背诵题。

批改规则：

- 全对：说明答对原因，标记当前点已掌握，然后进入下一小点
- 有错：只展开错题，解释误区、给正确答案、补一条排查提示，然后仍然回到主线继续推进
- 连续答错：先缩回最小例子或最小边界，再重新出题

## 七、统一练习模板

每个专项都尽量做这三类练习：

1. 正常路径
   - 写一个最小可运行版本
2. 失败路径
   - 故意制造一个典型错误，再修复它
3. 改造路径
   - 把当前实现改成更贴近真实系统的版本

## 八、统一复盘标准

每完成一个阶段，至少补一条复盘：

- 当前最容易混淆的概念是什么
- 哪个失败案例最有代表性
- 下次遇到类似问题时先查什么
- 哪个知识点已经能迁移到 AI 伴侣项目中

## 九、提问质量检查

发问前先补齐这几项，得到的回答会更有效：

- 当前学到哪一章
- 想理解概念、写代码、查 bug，还是做测验
- 贴最小代码，不要贴无关上下文
- 明确说明卡点：概念不懂、类型报错、运行报错、设计拿不准

## 十、专项之间怎么串起来

推荐用这条主线把分散知识接起来：

`LangChain / LangGraph → Monorepo → Hono.js → Zod → Vercel AI SDK / MCP → Next.js / React Query → Cloudflare / Drizzle ORM / Turborepo`

每学完一个专项，都回到系统视角追问：

- 它在完整产品里处于哪一层
- 与上下游怎么衔接
- 哪些问题必须靠它解决
- 哪些问题不该交给它解决
