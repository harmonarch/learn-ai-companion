# MCP 学习计划

## 这个领域解决什么问题

MCP 用统一协议连接模型与外部能力，让工具、资源、提示模板等能力以标准方式暴露出来，适合做可插拔工具系统和跨进程能力集成。

## 学习目标

- 理解 MCP Client / Server 的职责分工
- 能区分 tool、resource、prompt 等能力类型
- 能写一个最小 MCP server 或 client 调用链
- 能判断什么时候用 MCP，什么时候直接进程内调用即可
- 能把 MCP 放进 AI 伴侣的工具体系里

## 四阶段路径

### 阶段 1：入门理解
- Ch133
- 重点：MCP client、工具接入、协议调用
- 产出：能说清 client 为什么存在

### 阶段 2：核心能力
- Ch134
- 重点：MCP server、能力暴露、协议边界
- 产出：能写出最小 server 的资源或工具定义

### 阶段 3：实战落地
- 结合 Ch124-Ch125、Ch130、Ch135
- 重点：和 AI SDK、LangChain、LangGraph 的配合
- 产出：能把一个外部工具改造成 MCP 服务

### 阶段 4：巩固与排障
- 回看 client/server 边界和错误回传
- 重点：协议不兼容、工具描述不清、资源权限设计
- 产出：能定位接入失败属于 client、server 还是协议约定问题

## 核心易混点

- tool、resource、prompt 是不同能力类型，不该混成同一层。
- MCP 适合跨进程或跨服务暴露能力，不是所有工具都要接进去。
- client、server、协议层、业务层的职责要分开。
- MCP 和进程内工具的成本与收益不同。

## 各阶段过关标准

- 阶段 1：能解释它解决的问题，并写出或画出一个最小结构。
- 阶段 2：能独立完成核心能力的小例子，并说清执行流。
- 阶段 3：能把它接到当前 AI 伴侣项目的上下游，并解释边界。
- 阶段 4：能围绕典型错误定位根因，给出最小修复和复盘。

## 必做失败演练

- 故意把工具 schema 写得含糊，观察模型为什么不会正确使用工具。
- 故意制造 client 和 server 能力声明不匹配，定位失败发生在哪一段。
- 故意让错误返回不清晰，练习怎样补协议层可观测性。

## 改造练习

- 把一个本地工具改造成 MCP server 暴露的工具，再比较保留进程内实现和改成 MCP 的边界。

## 系统连接

- 上游常接 AI SDK、LangChain、LangGraph 的工具调用链。
- 下游常接外部服务、跨进程能力和资源系统。
- 在 AI 伴侣里常落在标准化工具暴露层和外部能力接入层。

## 常见误区

1. 以为 MCP 只是“远程函数调用”，忽略资源和协议边界
2. 工具 schema 描述太弱，模型难以正确使用
3. 所有工具都想做成 MCP，忽略维护成本
4. 不区分进程内工具和跨服务工具的成本差异
